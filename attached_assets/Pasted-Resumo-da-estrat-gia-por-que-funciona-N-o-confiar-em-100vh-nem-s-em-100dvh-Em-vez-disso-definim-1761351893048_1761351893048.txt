Resumo da estratégia (por que funciona)

Não confiar em 100vh nem só em 100dvh. Em vez disso definimos uma variável CSS --app-height com window.innerHeight (valor real) e atualizamos ela quando o teclado abre/fecha.

Header e input ficam fixos com position: fixed — mas a altura do container (onde as mensagens rolam) é controlada por --app-height para não “ser empurrada”.

Usar visualViewport (quando disponível) para detectar mudanças causadas pelo teclado e ajustar layout dinamicamente.

Forçar scroll ao fundo quando o campo recebe foco (resolve quando teclado empurra área).

Pequenos ajustes para safe-area insets (iPhone com notch) e fallback.

Código completo (HTML + CSS + JS) — cole no Replit e teste

Substitua classes/IDs pelo que você usa no projeto; é um exemplo autossuficiente.

    border-bottom: 1px solid #ddd;

<script>
/*
  SCRIPT: atualiza --app-height e ajusta comportamento quando teclado mobile abre
*/

const root = document.documentElement;
const messages = document.getElementById('messages');
const chatInput = document.getElementById('chatInput');
const inputWrapper = document.getElementById('inputWrapper');

function setAppHeight(h) {
  root.style.setProperty('--app-height', (h || window.innerHeight) + 'px');
}

/* Inicial */
setAppHeight();

/* 1) Ajusta quando a janela redimens. (Chrome android, desktop) */
window.addEventListener('resize', () => {
  setTimeout(()=> setAppHeight(window.innerHeight), 50);
});

/* 2) Use visualViewport quando disponível - é mais preciso no mobile */
if (window.visualViewport) {
  function onVVResize() {
    // altura visível da viewport (exclui teclado)
    const vvHeight = window.visualViewport.height;
    // definimos a altura do app como a altura visível (para evitar que header seja empurrado)
    root.style.setProperty('--app-height', vvHeight + 'px');

    // ajustar scroll e translacão do input caso necessário (iOS/Android)
    // compute offset entre window.innerHeight e visualViewport.height (teclado aberto)
    const offset = window.innerHeight - vvHeight;
    // mova input para cima do teclado com transform (suaviza bugs)
    inputWrapper.style.transform = offset > 0 ? `translateY(-${offset}px)` : 'translateY(0)';
  }
  window.visualViewport.addEventListener('resize', onVVResize);
  window.visualViewport.addEventListener('scroll', onVVResize);
}

/* 3) Ao focar o input, force scroll para o fim (assim as mensagens ficam visíveis acima do teclado) */
function scrollToBottom() {
  // rola até o final - com pequeno delay para permitir animação do teclado
  setTimeout(()=> {
    messages.scrollTop = messages.scrollHeight;
  }, 250);
}

chatInput.addEventListener('focus', () => {
  scrollToBottom();
});

chatInput.addEventListener('blur', () => {
  // remove transform ao fechar teclado
  inputWrapper.style.transform = '';
});

/* 4) Enviar: testar o comportamento e rolar */
document.getElementById('sendBtn').addEventListener('click', () => {
  const v = chatInput.value.trim();
  if (!v) return;
  const div = document.createElement('div');
  div.className = 'message';
  div.textContent = v;
  messages.appendChild(div);
  chatInput.value = '';
  scrollToBottom();
});

/* 5) fallback: definir altura inicial quando página carrega (importante para alguns browsers) */
window.addEventListener('load', () => setAppHeight(window.innerHeight));
</script>

</body>
</html>

Checklist — se ainda apresentar problema, verifique isto (faça cada item)

Meta viewport presente: <meta name="viewport" content="width=device-width,initial-scale=1,viewport-fit=cover">

Não usar height:100vh em containers principais. Use --app-height como acima.

Teste em navegação privada (cache às vezes mantém css antigo).

No iOS: Safari pode ignorar visualViewport em versões antigas; o fallback de resize e setTimeout cobre isso.

Remova transform/animation pesada no container .chat-messages — pode conflitar com scroll nativo.

Evite position: fixed em elementos dentro de iframes sem allow-same-origin — Replit preview roda em iframe, algumas restrições ocorrem. Se você está testando dentro do preview do Replit, abra o link Open in new tab (preview isolado) e teste no navegador do celular: isso faz diferença.

Se usar PWA ou WebView, o comportamento muda: testar no navegador padrão do Android/iOS.

Observações sobre Replit (problema comum)

O preview do Replit roda dentro de um iframe — o comportamento do teclado pode ser diferente dentro do app do Replit. Sempre teste abrindo o URL público do Replit em uma aba do navegador do celular (não pelo app do Replit).

Se você estiver testando via repl.co em um iframe dentro do app Replit, abra o link direto no Chrome/ Safari do celular.
